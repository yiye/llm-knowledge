# Product Agent Workbench: From "Text Editor" to "Business Orchestrator"
> **产品代号**: Workbench / The Product OS
> **核心定位**: 一个将「文档（沉淀知识）」与「Sticker（碎片想法）」转化为「实际业务动作」的 Agentic 工作台。
> **一句话愿景**: 让文档不仅是被阅读的静态文字，而是驱动业务流转的可执行代码 (Executable Documentation)。

---

## 1. 核心洞察 (The Core Insight)

### 1.1 痛点：死掉的知识
在传统工作流中（Notion/Google Docs），文档是业务的**终点**或**墓地**。
*   写完 PRD，需要人脑理解，再手动去配置后台、建 Jira、写代码。
*   写完会议纪要，需要人工去把 Todo 搬运到任务管理软件。
*   **文档是“死”的，Cursor 里的代码是“活”的**（因为有 LSP、有编译器、有 Terminal）。

### 1.2 机会：文档即代码 (Docs as Code/Context)
借用 Cursor 的哲学：**如果 IDE 能读懂代码并执行重构，那么 Workbench 也能读懂文档并执行业务。**
*   **Input**: PRD 文档 / 随手记的 Sticker
*   **Process**: Agent 理解上下文 + Tool Use
*   **Output**: 内部系统配置 API 调用 / 数据库 SQL / 自动化脚本 / Jira Ticket

---

## 2. 核心架构：三层知识流转

### Layer 1: The Context (知识沉淀层)
*   **Structured Docs (体系化知识)**: PRD、技术方案、团队规划。这是 Agent 的 "Long-term Memory"。
*   **Stickers (碎片化知识)**: 
    *   类似桌面便签，记录 "临时 Todo"、"灵光一现"、"口头需求"。
    *   **关键特性**: Sticker 不是废纸，它是 **Context Seed (上下文种子)**。Agent 可以一键将 Sticker "织入" 到正式文档中，或者转化为一个具体的 Task。

### Layer 2: The Agent Brain (意图理解层)
*   **Global Project Graph**: Agent 知道 `PRD-V1.2.md` 里的 "新增用户标签字段" 对应数据库里的 `users` 表，也对应后台配置系统的 `Tag Config` 页面。
*   **Active Listening**: 编辑器不再是被动等待输入，而是像 Cursor 的 Copilot 一样，预测你的意图（"监测到你修改了费率描述，是否需要生成 SQL 更新线上配置？"）。

### Layer 3: The Execution (业务执行层)
*   **Product Ops Automation**: 
    *   *场景*: 在文档里写下 "活动时间：2026-02-01 ~ 2026-02-07"。
    *   *动作*: Agent 自动调用内部运营后台 API，生成一个 "预配置单" 供确认。
*   **Ticket Orchestration**:
    *   *场景*: 会议纪要中记录 " @Frontend 需优化首屏加载"。
    *   *动作*: 自动在 Linear/Jira 创建工单，并附带会议上下文。

---

## 3. 典型工作流 (The "Alive" Workflow)

### 场景 A：从 PRD 到 系统配置
1.  **撰写**: 产品经理在 `Feature-Reward.md` 中定义了新的奖励策略："新用户注册送 500 积分，有效期 3 天"。
2.  **识别**: Workbench 的 Agent 高亮这段文字，并在侧边栏提示："检测到配置变更需求"。
3.  **执行**: 用户点击 "Generate Config JSON"。
4.  **验证**: Agent 生成 JSON 并调用测试环境 API 进行模拟。
5.  **结果**: 文档下方出现一个 ✅，表示 "已同步至测试环境"。

### 场景 B：Sticker 的生命周期
1.  **记录**: 也就是在吃饭时想到 "下个版本加个暗黑模式"，随手记在手机端 Sticker。
2.  **关联**: 回到电脑前，Workbench 提示 "有一条关于 UI 的 Sticker，是否关联到 `Design-Specs.md`？"
3.  **吸纳**: 点击确认，Agent 自动把这句话扩写成规范的 "UI Requirement" 插入文档，并将 Sticker 归档。

## 4. 竞品差异化 (Why not Notion AI?)

| 特性 | Notion AI | **Product Agent Workbench** |
| :--- | :--- | :--- |
| **定位** | **内容生成助手** (帮你写) | **业务执行代理** (帮你做) |
| **文档性质** | 用于阅读的文本 | **用于执行的指令** |
| **上下文** | Page 级别 (有限) | **Project 级别** (全知全能) |
| **输出物** | 更多的文字 | **API 请求、SQL、JSON、代码** |
| **交互** | 聊天对话框 | **行内 Diff / 浮动操作栏 (Action Bar)** |

---

## 5. 总结

这个 Idea 的本质是将 **DevOps 的理念引入 Product/Ops 领域**。
我们不再是在 "写文档"，我们是在 **"编写企业的业务逻辑"**。Cursor 让程序员用自然语言写代码，Product Workbench 让产品经理用自然语言配置企业。
