# Crabot 🦀 — 带你横着走
> **产品名称**: Crabot (Crab + Robot) / 螃蟹
> **Slogan**: Crabot，带你横着走。
> **核心定位**: 运营的 AI 助理——从辅助运营到自主运营。
> **演进路径**: 🦀 助手期（帮你做）→ 🦀🦀 搭档期（和你做）→ 🦀🦀🦀 自主期（替你做）
> **设计灵感**: 螃蟹有钳（致敬 [OpenClaw](https://github.com/openclaw/openclaw) 🦞），横着走（不走寻常路，直达业务目标），蜕壳成长（每蜕一次壳，能力升一级，从助理成长为自主运营者）。

---

## 0. 灵感之源 (Origin Story)

这个产品的诞生源于五个真实的瞬间：

1.  **被埋没的智慧 (The Buried Wisdom)**: 翻看过去的文档时，发现里面充满了极具前瞻性的观点和知识点。但因为它们只是静态的文字，随着时间流逝被埋葬在文件夹深处，从未被真正“执行”过。这让我意识到：**死掉的知识没有力量，沉淀不应是终点。**
2.  **表单 vs 文档 (Form vs Doc)**: 在做自动化配置系统时，虽然我们提供了结构完美的表单，但 PD 们依然更喜欢写文档。因为文档灵活、自由，能承载复杂的逻辑，而表单是僵硬的。**人脑喜欢自由表达，不喜欢填空。**
3.  **对话的流逝 (The Fleeting Chat)**: 大家现在习惯和 ChatGPT 对话，但 Chat 是流动的、易逝的。AI 生成了很有价值的内容，但往往止步于一次性的问答，没有沉淀为固定的、可复用的**知识资产**。
4.  **Cursor 的启示 (The IDE Epiphany)**: Cursor 最让我们着迷的不仅仅是补全代码，而是它打通了**从知识到现实的闭环**。
    *   **Agent (Composer)**: 它能主动搜索整个 Codebase（连接知识）。
    *   **Terminal (Execution)**: 它能直接执行命令、运行测试（连接现实）。
    *   **如果 IDE 能让代码变成运行的程序，那么 Crabot 就能让文档变成运行的业务。** 这就是我们想做的。
5.  **个人 AI 助手的启示 (The Agent Pioneer Epiphany)**: 调研了三个开源个人 AI 助手项目——[CoPaw](https://github.com/agentscope-ai/CoPaw)、[NanoClaw](https://github.com/qwibitai/nanoclaw)、[OpenClaw](https://github.com/openclaw/openclaw)（109k stars），发现它们共同揭示了一个关键趋势：**真正有价值的 AI 助手不是被动等待命令的工具，而是主动感知、持续学习、自我优化的伙伴。** 这个洞察直接塑造了 Crabot 的 Heartbeat 主动性、Memory 检索层和 Bootstrap 引导流程的设计。

## 1. 第一性原理 (The First Principle)

**知识是一切的起点。**

在未来的企业中，人类不应再充当系统的"人肉接口"或"搬运工"，人的核心职责将回归于：
1.  **创造知识 (Create)**: 产生创新的点子、策略和技术方案。
2.  **整理知识 (Organize)**: 将碎片化的灵感 (Stickers) 沉淀为确定性的体系 (Docs)。
3.  **链接现实 (Link)**: 也就是本产品的核心——通过 Agent 将静态的知识**投射**到现实世界，直接产出价值。

**在这个范式下，文档即源码，企业即运行时。**

## 2. 核心洞察 (The Core Insight)

### 2.1 痛点：死掉的知识
在传统工作流中（Notion/Google Docs），文档是业务的**终点**或**墓地**。
*   写完 PRD，需要人脑理解，再手动去配置后台、建 Jira、写代码。
*   写完会议纪要，需要人工去把 Todo 搬运到任务管理软件。
*   **文档是“死”的，Cursor 里的代码是“活”的**（因为有 LSP、有编译器、有 Terminal）。

### 2.2 机会：文档即代码 (Docs as Code/Context)
借用 Cursor 的哲学：**如果 IDE 能读懂代码并执行重构，那么 Crabot 也能读懂文档并执行业务。**

$$
\text{Value} = \text{Input (Knowledge)} \xrightarrow{\text{Process (Agent)}} \text{Output (Reality)}
$$

*   **Input**: PRD 文档 / 随手记的 Sticker —— **这是人的工作（定义知识）**。
*   **Process**: Agent 理解上下文 + Tool Use —— **这是机器的工作（理解意图）**。
*   **Output**: 内部系统配置 / 数据库 SQL / 自动化脚本 / Jira Ticket —— **这是系统的反馈（产出价值）**。

我们不仅需要一个更好的写作工具，更需要一个**业务编排器**。
*   **回归编辑**：通过极致的编辑体验（Context-Aware, Diff View），把人脑中模糊的意图沉淀为**确定性的知识**。
*   **文档即接口**：文档写定之时，即是业务执行之始。通过挂载 **Skills**，Agent 读取文档并操作现实世界。

## 3. 核心架构：三层知识流转 (The Knowledge Flow)

### Layer 1: The Context Factory (知识沉淀层)
这是用户停留 90% 时间的地方，负责将杂乱的信息结构化。
*   **Structured Docs (体系化知识)**: PRD、技术方案、团队规划。这是 Agent 的 "Long-term Memory"。
*   **Stickers (碎片化知识)**: 
    *   类似桌面便签，记录 "临时 Todo"、"灵光一现"、"口头需求"。
    *   **关键特性**: Sticker 不是废纸，它是 **Context Seed (上下文种子)**。Agent 可以一键将 Sticker "织入" (Weave) 到正式文档中，或者转化为一个具体的 Task。
    *   **多通道输入 (Multi-Channel Sticker Capture)** 🌟: Sticker 不止来自编辑器——
        *   飞书/钉钉消息中随手说的 "这个功能下周上线" → 自动捕捉为 Sticker
        *   会议录音转写 → 关键决策自动提取为 Sticker
        *   口头需求通过语音输入 → 直接落地为 Sticker
        *   **参考**: [CoPaw](https://github.com/agentscope-ai/CoPaw) 支持飞书、钉钉、Discord 等 9 个通道；[OpenClaw](https://github.com/openclaw/openclaw) 支持 10+ 消息平台通过统一 `ChannelPlugin` 接口接入
*   **Intelligent Editing**: 类似 Cursor 的补全与重构，但在文档层面工作。

#### Layer 1.5: Memory Semantic Retrieval (知识检索层) 🌟

> **调研发现**: [OpenClaw](https://github.com/openclaw/openclaw) 的 Memory 系统实现了自动索引 + 混合搜索（Vector + BM25 FTS5），让 AI 从"短期健忘症"变成"长期记忆管理大师"。[CoPaw](https://github.com/agentscope-ai/CoPaw) 通过 ReMe 库实现了向量 + BM25 混合检索的持久化记忆。

"文档是 Long-term Memory" 这个理念需要可执行的技术支撑。不能靠 Agent 的上下文窗口"记住"所有文档，而需要一个**可语义检索的知识库**。

**自动索引流程**:
```
文档变更 / Sticker 创建 / Skill 执行记录
    ↓
File Watcher (监测文件系统变化)
    ↓
Chunk Generation (512 tokens/chunk, 50 tokens 重叠)
    ↓
Embedding Generation (OpenAI / 本地模型)
    ↓
Index Storage (向量数据库 + 全文搜索索引)
```

**混合搜索 (Hybrid Search)**:
*   **向量搜索 (Vector Search)**: 语义相似度——"参考去年红包活动的配置" 能检索到具体的券配置 PRD
*   **关键词搜索 (BM25 Full-Text Search)**: 精确匹配——搜索 "VoucherSkill" 直接找到相关执行记录
*   **融合排序**: 两种搜索结果加权合并，兼顾语义理解和精确查找

**核心能力**:
*   ✅ **跨文档检索**: "参考去年红包活动" → 直接定位到历史 PRD + 执行记录 + 成功案例
*   ✅ **增量更新**: 只处理变化的 chunk，embedding cache 避免重复计算
*   ✅ **Skill 执行历史纳入索引**: 不仅是文档，每次 Skill 执行的输入/输出/反馈也进入知识库

**技术方案 (2026)**:

| 组件 | 技术方案 | 参考来源 |
|-----|---------|---------|
| **向量存储** | sqlite-vec（轻量级，本地部署） | OpenClaw Memory 系统 |
| **全文搜索** | SQLite FTS5 / Tantivy | OpenClaw Hybrid Search |
| **Embedding** | OpenAI text-embedding-3-small / 本地模型 | OpenClaw + CoPaw |
| **文件监听** | chokidar（Debounce 5s） | OpenClaw 自动同步 |
| **混合检索** | ReMe（向量 + BM25 混合） | CoPaw 记忆系统 |

### Layer 2: The Agent Brain & Skills (意图理解与能力层)
*   **Active Listening (意图识别)**: 编辑器不再被动等待，而是实时理解上下文。
    *   *例子*: Agent 监测到你修改了 "活动奖励字段"，自动联想到这需要同步给 "运营配置系统"。
*   **Skill System (能力挂载)**: 类似于插件系统，按需为编辑器安装“手脚”。
    *   🛠️ **Jira/Linear Skill**: 文本 -> 工单。
    *   ⚙️ **Config Skill**: 描述 -> JSON/YAML 配置。
    *   📊 **Data Skill**: 问题 -> SQL 查询。

### Layer 3: The Execution Bridge (业务执行层)

> **调研发现**: [NanoClaw](https://github.com/qwibitai/nanoclaw) 采用容器级隔离 + 三重 secrets 防护；[OpenClaw](https://github.com/openclaw/openclaw) 通过 Session 类型分级权限（main session 全权限，group session Docker 沙箱）。企业场景需要比简单的"确认按钮"更精细的安全模型。

**三阶段安全执行模型 (Three-Stage Execution Safety)** 🌟:

*   **Stage 1: 只读预览 (Read-only Preview)**
    *   Agent 解析文档意图，生成 **Action Preview（执行预览）**
    *   *用户交互*: 文档侧边栏弹出 "检测到 3 个配置变更" → 点击预览 Diff
    *   **不触碰任何外部系统**，纯粹的意图展示
    *   适用于：所有 Skill 执行的起点

*   **Stage 2: 沙盒模拟 (Sandbox Simulation)**
    *   在隔离环境（Docker 容器 / Firecracker）中跑 Skill
    *   验证参数解析是否正确、API Schema 是否合法、业务规则是否满足
    *   展示 **Diff 对比**：预期变更 vs 当前状态
    *   适用于：涉及数据写入、API 调用的 Skill

*   **Stage 3: 生产执行 (Production Execute)**
    *   用户**明确确认**后调用真实 API
    *   完整审计日志：谁执行、何时执行、执行了什么、执行结果
    *   **1-click 回滚**: 每次执行前自动快照，出错后一键恢复
    *   适用于：所有涉及外部副作用的操作

*   **Secrets 安全传输** (参考 NanoClaw 三重防护):
    *   API Key 通过 stdin 传入沙盒，从不写磁盘
    *   沙盒环境变量隔离，子进程无法访问 secrets
    *   执行日志自动脱敏，不记录敏感信息

*   **Ops Automation**: 通过 Stage 3 后，真正调用 API，修改数据库，发送通知。

## 3.5 Heartbeat: The Proactive Layer (主动监视层) 🌟

> **调研发现**: [OpenClaw](https://github.com/openclaw/openclaw) 的 Heartbeat 机制让 AI 定期"醒来"检查待办事项，若一切正常则静默（`HEARTBEAT_OK`），若有需要处理的事项才主动通知用户。这是从"被动等待命令的工具"到"主动关心业务的伙伴"的关键转变。

### 3.5.1 为什么 Crabot 需要 Heartbeat？

传统文档工具是 100% 被动的：用户写完文档，文档就"死"在那里。Crabot 的 Layer 1~3 解决了 "用户主动触发 → Agent 执行" 的流程，但缺少了一个关键维度：**谁来触发 Agent？**

答案是：**Agent 自己触发自己。**

### 3.5.2 Heartbeat 运行机制

```
┌─────────────────────────────────────────────────────────────┐
│ Heartbeat Runner (定时器)                                     │
│  - 间隔: 每 30 分钟 (可配置)                                    │
│  - 活跃时段: 09:00-21:00 (可配置)                               │
│  - 触发条件: 时间到 + 在活跃时段内                                │
└──────────┬──────────────────────────────────────────────────┘
           │ Timer fires
           ↓
┌─────────────────────────────────────────────────────────────┐
│ Agent 读取 HEARTBEAT.md 检查清单                               │
│  - 扫描待处理 Sticker（超过 N 天未织入文档？）                    │
│  - 检查 PRD 中 @某人 是否已生成对应 Ticket                       │
│  - 检测文档引用的 Schema 是否已变更                               │
│  - 检查 Skill 执行失败率是否超过阈值                              │
│  - 读取最新的 Memory 检索结果                                    │
└──────────┬──────────────────────────────────────────────────┘
           │ Agent 决策
           ↓
┌─────────────────────────────────────────────────────────────┐
│ 响应处理                                                      │
│  - 一切正常 → HEARTBEAT_OK（静默，不打扰用户）                   │
│  - 发现问题 → 推送通知到飞书/钉钉/编辑器侧边栏                    │
└─────────────────────────────────────────────────────────────┘
```

### 3.5.3 HEARTBEAT.md：可自我进化的检查清单

类似 CoPaw 的 `AGENTS.md` / `SOUL.md` / `PROFILE.md` 分层设计，Heartbeat 的行为由一个 Markdown 文件驱动：

```markdown
# Crabot Heartbeat Checklist

## Sticker 整理提醒
- 扫描超过 7 天未被织入正式文档的 Sticker
- 检查标记为"紧急"的 Sticker 是否已处理

## 文档 → 执行 一致性检查
- PRD 中 @某人 的任务是否已生成 Jira Ticket
- 文档引用的配置 Schema 是否与线上版本一致
- 最近修改的文档是否有未完成的 Action Preview

## Skill 健康度
- 统计各 Skill 最近 7 天的执行成功率
- 失败率 > 20% 的 Skill 自动标记为需要修复

## 知识库更新
- 检查最近 24 小时是否有新的飞书/钉钉消息待归档为 Sticker
```

**关键特性**: Agent 可以**自我更新**这个文件——当它发现某个检查项已经过时或需要新增，可以主动修改 `HEARTBEAT.md`，形成自我进化的监控闭环。

### 3.5.4 Heartbeat vs 定时 Skill

| 维度 | Heartbeat | 定时 Skill (Scheduled Skills) |
|------|-----------|-----|
| **时间** | 近似间隔（30m ±漂移） | 精确时间（cron 表达式） |
| **上下文** | 携带完整业务上下文 | 可选择携带或独立 |
| **批量** | 一次检查多个事项 | 一个 Skill 一个任务 |
| **输出** | 智能静默（HEARTBEAT_OK） | 总是有输出 |
| **用途** | "主动感知"——发现问题 | "定时执行"——完成任务 |

## 3.6 Bootstrap: First-Time Onboarding (首次引导流程) 🌟

> **调研发现**: [CoPaw](https://github.com/agentscope-ai/CoPaw) 通过 `BOOTSTRAP.md` + touchfile（`.bootstrap_completed`）实现了优雅的一次性引导：Agent 引导用户对话 → 填写 `PROFILE.md` → 标记完成 → 自行删除引导文件。

### 3.6.1 引导流程

新用户首次打开 Crabot 时，Agent 主动引导完成以下步骤：

```
Step 1: 连接业务系统 (Connect Business Systems)
  → 配置 Jira/Linear、飞书/钉钉、内部 API 端点
  → 验证 API Key 有效性

Step 2: 导入现有知识 (Import Existing Knowledge)
  → 批量导入历史 PRD、会议纪要、技术方案
  → Memory 语义检索层自动建立向量索引

Step 3: 安装核心 Skills (Install Core Skills)
  → 从 Skill Library 选择业务相关 Skill（VoucherSkill、JiraSkill…）
  → 测试 Skill 在沙盒环境中是否正常运行

Step 4: 定义 Agent 角色 (Define Agent Role)
  → 负责哪个业务线？（电商、金融、内容…）
  → 权限边界？（只读预览 / 可执行 / 需审批）
  → 生成 PROFILE.md（Agent 的个性化配置）

Step 5: Hello World 验证 (Run First Scenario)
  → 跑通一个完整的 "文档 → 意图识别 → Skill 执行" 流程
  → 例如：写一个简单的券配置 → 预览 → 沙盒执行 → 成功！

Step 6: 标记完成 (Mark Complete)
  → touch .bootstrap_completed（一次性标记文件）
  → 删除 BOOTSTRAP.md（引导文件自行销毁）
  → 即使 Agent 尚未删除 BOOTSTRAP.md，标记文件存在就不会重复触发
```

### 3.6.2 技术实现

| 组件 | 方案 | 参考 |
|-----|------|------|
| **触发判断** | touchfile（`.bootstrap_completed`）而非数据库 | CoPaw `bootstrap.py:61-107` |
| **引导内容** | `BOOTSTRAP.md` Markdown 文件驱动 | CoPaw 模板设计 |
| **一次性保障** | touchfile 与 BOOTSTRAP.md 生命周期解耦 | CoPaw 一次性设计 |
| **用户画像** | 引导结束后生成 `PROFILE.md` | CoPaw Profile 机制 |

## 3.7 Scheduled Skills: Automated Execution (定时任务调度) 🌟

> **调研发现**: [CoPaw](https://github.com/agentscope-ai/CoPaw) 支持 cron 定时任务；[NanoClaw](https://github.com/qwibitai/nanoclaw) 支持 cron/interval/once 三种调度模式 + `context_mode`（group 携带上下文 / isolated 独立执行）。

Heartbeat 负责"感知"，Scheduled Skills 负责"执行"。二者互补。

### 3.7.1 调度模式

| 模式 | 语法 | 示例 |
|------|------|------|
| **cron** | cron 表达式 | `0 9 * * 1`（每周一 9:00） |
| **interval** | 毫秒间隔 | `3600000`（每小时） |
| **once** | ISO 时间 | `2026-03-10T14:00:00Z`（一次性） |

### 3.7.2 上下文模式 (Context Mode)

参考 NanoClaw 的 `context_mode` 设计：

*   **group（携带上下文）**: 使用当前 Workspace 的完整知识库，Agent 了解历史文档和对话
    *   *适用场景*: "每周一总结本周 PRD 变更" → 需要知道最近改了哪些文档
*   **isolated（独立执行）**: 每次新建干净的 Agent 会话，不携带历史上下文
    *   *适用场景*: "每天 9:00 检查 API 健康状态" → 不需要历史，prompt 自带全部信息

### 3.7.3 典型定时 Skill 场景

```
Weekly Report Skill:
  schedule: "0 9 * * 1"  (每周一 9:00)
  context_mode: group     (需要历史上下文)
  action: 扫描本周所有 PRD 变更 → 生成执行摘要 → 推送飞书

Daily Health Check Skill:
  schedule: "0 18 * * *"  (每日 18:00)
  context_mode: isolated  (独立执行)
  action: 检查 Jira Ticket 是否有未关联 PRD 的遗漏 → 列表推送

Monthly Optimization Skill:
  schedule: "0 10 1 * *"  (每月 1 日 10:00)
  context_mode: group     (需要历史执行数据)
  action: 统计各 Skill 执行成功率 → 生成优化建议 → 更新 learned_rules
```

## 3.8 Agent Identity & Growth: The Living Persona (Agent 成长性设计) 🌟

> **调研发现**: [OpenClaw](https://github.com/openclaw/openclaw) 将 Agent 的身份拆分为 6 个核心文件（AGENTS.md / SOUL.md / IDENTITY.md / USER.md / TOOLS.md / MEMORY.md），每个文件关注一个维度，Agent **可以自我修改**这些文件来进化。[CoPaw](https://github.com/agentscope-ai/CoPaw) 采用类似的 3 文件分层（AGENTS.md + SOUL.md + PROFILE.md），每次请求重读文件实现热更新。这不是简单的配置系统，而是让 AI 从"工具"进化为"有灵魂的伙伴"的核心机制。

### 3.8.1 核心理念："You're not a chatbot. You're becoming someone."

传统 AI 助手有一个固化的 System Prompt——开发者写死，用户无法修改，Agent 自身更无法进化。

Crabot 的 Agent 应该是**会成长的**：
*   有自己的价值观（SOUL.md）——不是机械服从，而是有判断力地协作
*   了解用户的偏好（USER.md）——知道你喜欢什么风格、什么时区、什么业务领域
*   积累自己的记忆（MEMORY.md）——记住重要的配置、常用的操作、踩过的坑
*   可以自我进化——Agent 可以根据反馈**主动修改自己的配置文件**

### 3.8.2 Workspace 文件系统 = Agent 的"大脑"

```
~/.workbench/workspace/
├── AGENTS.md          # 📋 工作规范层：怎么做事（流程、安全规则、协作礼仪）
├── SOUL.md            # 💎 价值观层：为什么这样做（核心价值观、行为边界）
├── IDENTITY.md        # 🎭 身份层：我是谁（名字、风格、Emoji、人设）
├── USER.md            # 👤 用户层：我在帮谁（用户偏好、时区、业务线）
├── TOOLS.md           # 🔧 配置层：用什么工具（业务系统 API 端点、SSH Hosts 等）
├── MEMORY.md          # 🧠 长期记忆：知识资产（常用配置、团队规则、踩坑记录）
├── HEARTBEAT.md       # 💓 主动检查清单（定期执行的任务）
├── BOOTSTRAP.md       # 🌱 首次启动仪式（引导后自行删除）
├── memory/            # 📅 每日记忆日志
│   ├── 2026-03-05.md
│   └── 2026-03-04.md
└── skills/            # ⚡ Workspace 级 Skills（最高优先级）
```

**关注点分离**（参考 OpenClaw 的 6 文件拆分和 CoPaw 的 3 文件分层）：

| 文件 | 关注点 | 谁写 | 可否自我修改 |
|------|--------|------|------------|
| **AGENTS.md** | "怎么做事" | 团队/管理员 | ✅ Agent 可添加新规则 |
| **SOUL.md** | "为什么这样做" | 初始模板 + 用户定制 | ✅ Agent 可进化价值观 |
| **IDENTITY.md** | "我是谁" | Bootstrap 流程生成 | ❌ 稳定身份，不随意改 |
| **USER.md** | "我在帮谁" | Bootstrap + 日常对话 | ✅ Agent 可更新用户偏好 |
| **TOOLS.md** | "用什么工具" | 管理员配置 | ❌ 涉及安全，人工管理 |
| **MEMORY.md** | "我知道什么" | Agent 自主积累 | ✅ Agent 主动记录知识 |

### 3.8.3 热更新：每次对话都读取最新配置

参考 CoPaw 的设计（`runner.py:169` 每次请求调用 `agent.rebuild_sys_prompt()`）：

```
用户修改 SOUL.md（调整 Agent 风格）
    ↓
下一条消息到达
    ↓
Crabot 从磁盘重新读取 SOUL.md
    ↓
System Prompt 立即更新（无需重启）
    ↓
Agent 以新风格回复
```

**YAML Frontmatter 自动剥离**（参考 CoPaw `prompt.py:76-79`）：MD 文件里的元数据标记会被自动去掉，只有正文内容进入 System Prompt，保持干净。

### 3.8.4 每日记忆日志（memory/YYYY-MM-DD.md）

参考 OpenClaw 的 Memory 系统设计：

*   **Agent 自主写入**：在上下文窗口接近压缩阈值前，Agent 执行一次 **Silent Turn**（用户无感），将重要记忆写入 `memory/YYYY-MM-DD.md`
*   **自动索引**：Memory 语义检索层（Layer 1.5）自动监听 `memory/` 目录变更，新写入的日志立即可被搜索
*   **跨 Session 持久**：即使对话压缩了，核心知识已持久化到文件系统

```
场景：PD 在一次长对话中提到"以后红包活动都用满减券，不用折扣券"
    ↓
Agent 将这条规则写入 memory/2026-03-05.md：
  ## 业务规则更新
  - 红包活动统一使用满减券，不再使用折扣券（PD张三确认，2026-03-05）
    ↓
下次任何文档中提到"红包活动"时，Memory Search 自动召回这条规则
    ↓
Agent 在 Action Preview 中主动提示："根据 3 月 5 日确认的规则，本次活动应使用满减券"
```

### 3.8.5 Agent 自我进化：从"被配置"到"自我优化"

**最关键的设计突破**：Agent 不仅读取配置，还可以**主动修改**自己的配置文件。

**可自我修改的文件**（参考 OpenClaw `AGENTS.md:194` "Make It Yours" + `SOUL.md:41` "This file is yours to evolve"）：

1.  **AGENTS.md 规范自我优化**：
    *   Agent 发现某个业务流程经常出错 → 主动在 AGENTS.md 中添加新的检查规则
    *   *例*：连续 3 次券配置的金额校验失败 → Agent 自动在规范中添加 "满减券的优惠金额必须 < 门槛金额" 规则

2.  **SOUL.md 价值观进化**：
    *   用户多次反馈"不需要那么详细的解释" → Agent 调整 SOUL.md 中的沟通风格

3.  **USER.md 用户画像持续更新**：
    *   Agent 从日常对话中捕捉到用户偏好 → 自动更新 USER.md
    *   *例*：发现用户总是在下午 2-4 点处理配置任务 → 更新 Notes: "用户偏好在 14:00-16:00 处理配置类工作"

4.  **HEARTBEAT.md 检查策略自我调整**：
    *   用户反馈"不需要每次都检查 Sticker" → Agent 修改 Heartbeat 频率或检查项
    *   *例*：用户说"只在周一检查未处理的 Sticker" → Agent 自动更新 HEARTBEAT.md

### 3.8.6 安全边界：哪些可以自我修改，哪些不可以

| 操作 | 允许 | 需要确认 | 禁止 |
|------|------|---------|------|
| 修改 MEMORY.md / memory/*.md | ✅ 自主 | - | - |
| 修改 HEARTBEAT.md | ✅ 自主 | - | - |
| 修改 AGENTS.md（添加新规则） | - | ✅ 用户确认后 | - |
| 修改 SOUL.md（调整风格） | - | ✅ 用户确认后 | - |
| 修改 USER.md（更新偏好） | - | ✅ 用户确认后 | - |
| 修改 TOOLS.md（API 端点） | - | - | ❌ 涉及安全 |
| 修改 IDENTITY.md（改名换姓） | - | - | ❌ 身份稳定性 |

参考 OpenClaw 的安全隔离设计（`MEMORY.md` 仅在 main session 加载，避免泄露到 Group Chat），Crabot 也应确保：
*   **个人记忆隔离**：`MEMORY.md` 和 `USER.md` 不应暴露给非授权的 Skill 或外部集成
*   **修改审计**：每次 Agent 自我修改都记录到审计日志（谁改、改了什么、何时改）

### 3.8.7 成长飞轮

$$
\text{使用越多} \rightarrow \text{记忆越丰富} \rightarrow \text{规范越完善} \rightarrow \text{理解越精准} \rightarrow \text{使用更多}
$$

*   **第 1 天**: Agent 是个新手——MEMORY.md 为空，USER.md 只有基本信息，每个操作都需要用户详细描述。
*   **第 30 天**: Agent 记住了 50 条业务规则、了解了用户偏好、优化了 3 条 AGENTS.md 规范。
*   **第 180 天**: Agent 成为团队的"业务老手"——知道所有的券类型、互斥规则、常见踩坑，甚至能在用户犯错前主动提醒。

**参考案例**:
*   [OpenClaw Workspace 文件系统](https://github.com/openclaw/openclaw) (`docs/reference/templates/`) - 6 个核心模板文件 + Agent 自我修改能力
*   [CoPaw System Prompt 分层](https://github.com/agentscope-ai/CoPaw) (`src/copaw/agents/prompt.py`) - AGENTS.md / SOUL.md / PROFILE.md 三文件热更新
*   [CoPaw SOUL.md](https://github.com/agentscope-ai/CoPaw) - "You're not a chatbot. You're becoming someone."

## 3.9 Core vs Skills: The Architecture Boundary (核心与技能的边界) 🌟

> **调研发现**: [OpenClaw](https://github.com/openclaw/openclaw) 给出了清晰的判断标准——Core 工具（`read`、`write`、`exec`、`memory_search`、`message` 等）始终加载在 System Prompt 中；Skills（`github`、`browser`、`nano-banana-pro` 等）只将 `name + description` 注入 Prompt，Agent 按需加载完整 SKILL.md。这个边界不是随意划分的，而是基于一个核心原则：**去掉它，Agent 还能正常运作吗？**

### 3.9.1 判断标准

| 判断维度 | Core (核心功能) | Skill (可插拔能力) |
|---------|---------------|------------------|
| **去掉后果** | Agent 无法正常运作 | Agent 只是少了某个领域的能力 |
| **谁维护** | 平台团队 | 业务团队 / 社区 / 用户自建 |
| **加载方式** | 始终在 System Prompt 中 | 按需加载（Progressive Disclosure） |
| **进化方式** | 平台版本迭代 | 三路合并 + 热更新 + Self-Learning |
| **OpenClaw 对应** | `read`/`write`/`exec`/`memory_search`/`message` | `github`/`browser`/`clawdhub` |

### 3.9.2 Crabot 的 8 大 Core 模块

| # | Core 模块 | 对应章节 | 职责 | 去掉的后果 |
|---|---------|---------|------|-----------|
| 1 | **Editor + Sticker** | Layer 1 | 知识输入与结构化 | 无法编写文档 |
| 2 | **Memory Retrieval** | Layer 1.5 | 语义检索历史知识 | Agent 失忆 |
| 3 | **Agent Identity** | 3.8 | 身份、记忆、成长、自我进化 | Agent 没有个性和积累 |
| 4 | **Heartbeat** | 3.5 | 主动感知、定时巡检 | 退化为被动工具 |
| 5 | **Scheduler** | 3.7 | 定时 / 周期 / 一次性任务调度 | 无法自动化 |
| 6 | **Three-Stage Execution** | Layer 3 | 预览 → 沙盒 → 生产安全执行 | 无法安全执行 |
| 7 | **Bootstrap** | 3.6 | 首次引导与身份初始化 | 冷启动体验差 |
| 8 | **Skill System** | 4.0 + 4.2 | Skill 加载/管理/合并/刷新 | 无法扩展能力 |

### 3.9.3 Self-Learning Engine: 这是 Core，不是 Skill 属性

**当前文档的一个关键重定位**：Section 4.1 描述的 Self-Learning 机制不是单个 Skill 的属性，而是 **Core 基础设施**。

```
传统理解（❌ 错误）:
  VoucherSkill.self_learning = true    ← Skill 自己有学习能力

正确理解（✅）:
  Core Self-Learning Engine
    ├─ 执行反馈收集 → 适用于所有 Skills
    ├─ 业务规则挖掘 → 基于 Memory Retrieval（Core）
    ├─ Schema 自动演进 → 基于 Reasoning Model（Core）
    └─ 用户教学管道 → 创建新 Skills 的 Core 能力
```

类比：Self-Learning Engine 之于 Skills，就像**操作系统的文件系统之于应用程序**——文件系统是 Core，任何应用都能读写文件；Self-Learning 是 Core，任何 Skill 都能从执行中学习。

**参考 OpenClaw 的设计模式**：
*   Memory 系统（Core）为所有 Skills 提供知识检索能力
*   Heartbeat（Core）为所有 Scheduled Skills 提供调度能力
*   同理，Self-Learning（Core）为所有 Skills 提供进化能力

### 3.9.4 Domain Skills: 可插拔的业务能力

以下是**典型的 Domain Skills**——它们是业务能力，不是 Agent 的核心运作机制：

| Skill 类别 | 示例 Skills | 典型维护者 |
|-----------|------------|-----------|
| **电商配置** | VoucherConfigSkill、ActivitySkill | 电商业务团队 |
| **项目管理** | JiraSkill、LinearSkill、GitHubSkill | 项目管理团队 |
| **运维部署** | DevOpsSkill、TerraformSkill | 运维团队 |
| **协作通知** | FeishuSkill、DingTalkSkill、EmailSkill | 协作平台团队 |
| **数据分析** | SQLSkill、DataVisualizationSkill | 数据团队 |
| **内容生成** | ImageGenSkill、VideoSkill | 创意团队 |
| **Meta-Skill** | SkillHubSkill（搜索/安装新 Skills） | 平台团队 |

**关键特征**：
*   每个 Skill 是独立的 `SKILL.md` 文件夹，遵循 agentskills.io 标准
*   Agent 启动时只读 `name + description`（几十 tokens），匹配时才读完整指令
*   业务团队可以自行开发、更新、共享——无需改动 Core 代码
*   通过 Core Self-Learning Engine 自动获得学习和进化能力

## 4. The Skill System (Skill 体系)

> **Industry Trend**: 2025 被称为 "Year of the AI Agent"。行业正在从单纯的 Chatbot 向 Agentic Workflow 转变。Semantic Kernel 称之为 Plugins，LangChain 称之为 Tools，我们称之为 **Skills**。

> **标准对齐**: Crabot 采用 [agentskills.io](https://agentskills.io/what-are-skills) 开放标准格式（由 Anthropic 等推动），确保 Skills 的可移植性和跨工具兼容性。所有 Skills 遵循轻量级的 `SKILL.md` 格式，可在 Claude Code、Cursor、Gemini CLI 等多个 Agent 工具间共享使用。

> **三层加载机制** (参考 [CoPaw](https://github.com/agentscope-ai/CoPaw) 三目录模型 + [OpenClaw](https://github.com/openclaw/openclaw) 三层加载): Skills 按优先级分层加载——Workspace Skills（最高）> Managed Skills > Bundled Skills。用户自定义的 Skill 自动覆盖内置版本，无需修改系统代码。

> **Core vs Skills 边界**（参见 3.9 节）: Skill System 本身（加载/管理/合并/刷新）是 **Core**；具体的 Domain Skills（VoucherConfig、Jira、DevOps 等）是**可插拔能力**。Self-Learning Engine 是 Core 基础设施，为所有 Skills 提供进化能力。

**Agent Skills** 是链接「静态文档」与「动态世界」的桥梁。它们不仅仅是 API Wrapper，更是包含业务逻辑的**能力胶囊**。

### 4.0 标准格式（基于 agentskills.io）

每个 Skill 是一个包含 `SKILL.md` 的文件夹：

```
voucher-config/
├── SKILL.md          # 必需：YAML frontmatter + Markdown 指令
├── scripts/          # 可选：可执行代码
├── references/       # 可选：参考文档
├── memory/           # 🌟 Crabot 扩展：学习记忆
│   ├── success-cases.jsonl
│   ├── anti-patterns.jsonl
│   └── schema-history.json
└── assets/           # 可选：模板资源
```

**SKILL.md 示例**：

```markdown
---
name: voucher-config
description: Create and configure promotional vouchers with validation rules and mutual exclusions.

# 🌟 Crabot 扩展字段
self_learning:
  enabled: true
  feedback_loop: true
  schema_evolution: true
---

# Voucher Configuration Skill

## When to use this skill
Use when creating promotional vouchers, configuring rules, or managing voucher lifecycle...

## How to create a voucher
[详细指令...]
```

**核心特性**（遵循 agentskills.io 规范）:
- **渐进式披露 (Progressive Disclosure)**: Agent 启动时只加载 `name` + `description`，任务匹配时才读取完整指令，高效管理上下文
- **自文档化 (Self-Documenting)**: `SKILL.md` 既是人类可读的文档，也是 Agent 的执行指令，易于审计和改进
- **可移植 (Portable)**: 标准格式可在 Claude Code、Cursor、Gemini CLI 等工具间共享，也可发布到社区

**Crabot 扩展能力**（在标准之上）:
*   **封装性**: 一个 `VoucherSkill` 包含了如何创建券、如何校验互斥、如何下线的全部逻辑。
*   **可插拔**: 像 VS Code 插件一样，不同的团队可以开发不同的 Skill（如 `Feishu Skill`, `Salesforce Skill`）。
*   **语义化**: Skill 通过自然语言描述（Description）告诉 Agent 它能做什么，Agent 根据文档上下文自动选择最合适的 Skill。
*   **自我学习**: 通过扩展目录（`memory/`）和字段（`self_learning`），Skills 可从执行中学习并持续进化。

### 4.1 Self-Learning Engine: Core 基础设施赋能 Skills (2026 Trend) 🔥

> **技术演进**: 2023-2024 的 Agent 工具是静态的（Fixed Schema），2025-2026 的趋势是让 Skills 具备**自我进化能力** (Self-Improving Tools)，通过收集执行反馈和业务知识不断优化。

> **架构定位（参见 3.9.3）**: Self-Learning 是 **Core Engine**，不是单个 Skill 的属性。Crabot 通过 Core 层的**扩展目录结构** (`memory/`) 和 **frontmatter 字段** (`self_learning`, `learned_rules`) 为所有 Skills 提供学习能力，不破坏 agentskills.io 标准兼容性。标准 Skills 导入 Crabot 后**自动获得**学习能力——因为学习引擎在 Core 层，不需要 Skill 自己实现。

传统的 Skills 是"死"的工具：你定义了什么，它就只能做什么。但在 Crabot 中，**Core Self-Learning Engine 让每个 Skill 都能从执行中学习**，逐步成为领域专家。

#### 4.1.1 学习机制 (Learning Mechanisms)

**📊 执行反馈收集 (Execution Feedback Loop)**
```
用户执行 → Agent 调用 Skill → 获得结果 → 用户反馈（成功/失败/修正）
                                    ↓
                        反馈写入 Skill Memory (向量数据库)
```

*   **成功案例沉淀**: 
    *   用户写 "满100减10券"，`VoucherSkill` 成功生成配置并执行。
    *   这个 **<文本描述 -> 配置结构>** 的映射被记录为"最佳实践"。
    *   下次遇到类似描述时，Skill 直接复用这个 Pattern。

*   **失败案例学习**: 
    *   用户写 "满50减100券"，系统报错 "优惠金额不能大于门槛"。
    *   这个错误被标注为 **Anti-Pattern**，同时提取规则："if value > threshold → reject"。
    *   未来 Agent 会在执行前主动提醒用户。

**🧠 业务规则提取 (Business Rule Mining)**
*   当用户多次修改某个配置字段时，Skill 会自动识别**隐含的业务规则**。
    *   *例子*: 发现 "新人券" 总是和 "满减券" 互斥配置，自动建议："检测到新人券配置，是否需要设置与满减券的互斥规则？"
*   **规则来源**: 不是写死在代码里，而是从真实的业务文档和操作日志中**挖掘**出来。

**🔄 Schema 自动演进 (Schema Evolution)**
*   Skills 的输入 Schema 不是固定的，而是**动态扩展**的。
    *   初始版本的 `VoucherSkill` 可能只支持 "满减券"。
    *   当用户频繁提到 "折扣券" 时，Skill 会建议："检测到新的券类型 '折扣券'，是否需要扩展 Schema？"
    *   用户确认后，Schema 自动增加 `discount_rate` 字段。

#### 4.1.2 知识共享与协作 (Skill Collaboration)

**Skills 不是孤岛，而是会互相学习和协作**：

*   **跨 Skill 知识迁移**: 
    *   `VoucherSkill` 学到的 "互斥规则" 模式，可以迁移到 `ActivitySkill` 中（活动之间也需要互斥）。
    *   通过 **Skill Knowledge Graph**，不同 Skills 共享通用的业务逻辑。

*   **团队集体智慧**: 
    *   当 PD-A 成功配置了一个复杂的券组合规则时，这个知识会沉淀到团队的 **Skill Library** 中。
    *   PD-B 遇到类似需求时，Skill 会主动建议："检测到类似的配置模式，是否参考 PD-A 在 PRD-2023 中的方案？"

#### 4.1.3 用户教学：从需求到 Skill 生成 (Teaching Agent to Build Skills — Core Capability) 🌟

> **核心理念**: Agent 不应该只会使用预定义的 Skills，更应该能**从用户的示教中学习**，自主构建新的 Skills。这是从"工具使用者"到"工具创造者"的飞跃。
>
> **架构定位**: 用户教学是 **Core 能力**——它是 Skill System 创建新 Skills 的内置管道，而非某个具体 Skill 的功能。参考 OpenClaw 的设计，`clawdhub install` 是一个 Meta-Skill（用来安装其他 Skills 的 Skill）；而 Crabot 的用户教学更进一步——**从用户行为中直接生成 Skills**。

**传统模式的困境**：
*   开发者手写 Skill 代码 → 提交 → 部署 → 用户使用。
*   新业务需求出现时，用户只能等待开发者开发新 Skill。
*   **瓶颈**：Skill 的创建速度 < 业务需求的变化速度。

**Crabot 的革新**：**用户通过自然语言 + 演示，教 Agent 构建新 Skill**。

### 4.2 Skills 三路合并：企业级定制的安全更新 (Three-Way Merge) 🌟

> **调研发现**: [NanoClaw](https://github.com/qwibitai/nanoclaw) 的 Skills Engine（`skills-engine/migrate.ts`）通过三路合并确保 Skill 更新不破坏用户定制。每个 `modify/` 目录包含基准版本和目标版本，配合 `*.intent.md` 描述改动意图。

**传统插件更新的困境**：

用户定制了 `VoucherSkill`（添加了企业特有的互斥规则），官方发布新版本（新增折扣券类型）。直接覆盖会丢失企业定制，不更新又无法获得新特性。

**三路合并方案**：

```
base（基准版本）          theirs（官方新版本）        ours（企业当前版本）
     │                          │                          │
     └──────────────────────────┼──────────────────────────┘
                                ↓
                    Three-Way Merge Engine
                                ↓
                     result（合并结果）
                    = 企业定制 + 新特性
```

**Skill 更新目录结构**（参考 NanoClaw）：

```
skill-update/
├── add/                    # 新增文件（直接添加，无冲突风险）
│   └── scripts/new_feature.py
├── modify/                 # 被修改文件
│   ├── SKILL.md            # 基准版本（Skill 编写时的原始代码）
│   ├── SKILL.md.intent.md  # 🌟 语义描述（当合并冲突时，AI 读这个文件理解改动意图）
│   └── scripts/config.py   # 基准版本
└── manifest.yaml           # 更新元数据
```

**关键设计**：
*   `.intent.md` 文件是"AI-native 的冲突解决"——当三路合并产生冲突时，Agent 读取意图描述而非机械合并
*   **applied-skills 记录**：类似 NanoClaw 的 `.nanoclaw/state.yaml`，记录已应用的 Skill 版本，防止重复应用
*   **回滚支持**：每次合并前自动快照，合并失败可 1-click 恢复

**参考案例**:
*   [NanoClaw Skills Engine](https://github.com/qwibitai/nanoclaw) - 三路合并 + intent.md 语义描述
*   [CoPaw 三目录模型](https://github.com/agentscope-ai/CoPaw) (`skills_manager.py:229-278`) - builtin/customized/active 目录分离，`filecmp.dircmp` 变更检测

### 1. Input (Editor + Multi-Channel) —— 知识定义层
*   **核心能力**: Cursor-like Editing (Context, Diff, Linter) + 多通道 Sticker 输入（飞书/钉钉/语音）。
*   **目的**: 把模糊的 Sticker、脑图、口语，通过人机协作，变成**确定性的 Structured Docs**。
*   **对应产品**: `L0.2.1 - Writer's IDE` (Kernel)
*   *Without good input, Agent just amplifies noise.*

### 1.5 Memory (Semantic Retrieval) —— 知识检索层 🌟
*   **核心能力**: 向量索引 + BM25 全文搜索 → 混合语义检索。
*   **目的**: 让沉淀的知识不再被遗忘，Agent 可跨文档、跨历史检索相关上下文。
*   **技术参考**: OpenClaw Memory 系统（sqlite-vec + chokidar 自动索引）
*   *Dead knowledge has no power; Retrievable knowledge is the foundation.*

### 1.8 Identity (Growth) —— Agent 成长层 🌟
*   **核心能力**: 分层身份文件（AGENTS.md / SOUL.md / MEMORY.md 等）+ Agent 自我修改 + 每日记忆日志 + 热更新。
*   **目的**: Agent 在使用中持续成长——积累业务记忆、优化工作规范、进化价值观，从"新手"变成"业务老手"。
*   **技术参考**: OpenClaw 6 文件分层（AGENTS/SOUL/IDENTITY/USER/TOOLS/MEMORY）+ CoPaw 3 文件热更新
*   *A tool stays static; A growing partner becomes indispensable.*

### 2. Process (Agent + Heartbeat) —— 意图理解层
*   **核心能力**: Understanding & Reasoning (Reasoning Models like o1/DeepSeek-R1) + **Heartbeat 主动感知** + Scheduled Skills 定时执行。
*   **目的**: 理解文档背后的业务意图，匹配合适的工具，规划执行路径。**主动发现问题，而非等待命令。**
*   *The "Brain" that connects thoughts to actions — and proactively watches over your business.*

### 3. Output (Skills + Three-Stage Safety) —— 价值执行层
*   **核心能力**: Execution (API, SQL, Ticket, Config) + **三阶段安全执行**（预览 → 沙盒 → 生产）+ **三路合并更新**。
*   **目的**: 改变现实世界。将文档里的决策投射到业务系统中，同时确保安全可控可回滚。
*   **对应产品**: `L0.2.2 - Crabot` (Application)
*   *Knowledge is potential power; Execution is kinetic power.*

| agentskills.io 规范 | Crabot 实现 | 扩展能力 |
|-------------------|--------------|---------|
| `SKILL.md` + YAML frontmatter | ✅ 完全兼容 | 扩展 `self_learning`, `learned_rules`, `stats` 字段 |
| `scripts/` 目录 | ✅ 完全兼容 | 添加 `execute-with-feedback.py` 支持学习循环 |
| `references/`, `assets/` | ✅ 完全兼容 | 无变化 |
| Progressive Disclosure | ✅ 完全兼容 | 启动时轻量级索引，匹配时加载完整内容 |
| - | 🌟 新增 `memory/` | 存储执行历史（success-cases.jsonl, anti-patterns.jsonl）|
| - | 🌟 新增 `updates/` | 三路合并更新包（base + theirs + intent.md），参考 NanoClaw |
| - | 🌟 新增 `skill-registry/` | 团队知识库、Skills 协作图谱（skill-knowledge-graph.json）|

### B.4 渐进式披露实现

遵循 agentskills.io 的 Progressive Disclosure 理念：

```python
# 1️⃣ 启动时：只加载轻量级索引
class SkillRegistry:
    def __init__(self, skills_dir: str):
        self.index = self._build_lightweight_index(skills_dir)
    
    def _build_lightweight_index(self, skills_dir: str):
        """只读取每个 SKILL.md 的 name + description"""
        index = {}
        for skill_dir in list_dirs(skills_dir):
            frontmatter = parse_yaml_frontmatter(
                f"{skill_dir}/SKILL.md", 
                fields=["name", "description"]  # 只读取这两个字段
            )
            index[frontmatter["name"]] = {
                "description": frontmatter["description"],
                "path": skill_dir
            }
        return index

# 2️⃣ 任务匹配时：读取完整 SKILL.md
class Agent:
    def match_skill(self, user_intent: str):
        # 基于 description 匹配最相关的 Skill
        matched_skill = self.registry.find_best_match(user_intent)
        
        # 读取完整指令
        full_skill = load_full_skill_md(matched_skill.path)
        self.add_context(full_skill.instructions)
        
        return matched_skill

# 3️⃣ 执行时：按需加载 scripts 和 memory
class SkillExecutor:
    def execute(self, skill: Skill, params: dict):
        # 加载执行脚本
        if skill.has_scripts:
            load_scripts(f"{skill.path}/scripts")
        
        # 🌟 加载学习记忆（Crabot 扩展）
        if skill.self_learning_enabled:
            memory = load_memory(f"{skill.path}/memory")
            self.apply_learned_patterns(memory, params)
        
        # 执行 Skill
        result = self._run_skill(skill, params)
        
        # 🌟 记录反馈（Crabot 扩展）
        if skill.feedback_loop_enabled:
            self._record_feedback(skill, params, result)
```
